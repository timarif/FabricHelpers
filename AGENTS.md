# AGENTS.md — guidance for coding agents working on FabricHelpers

This file is read by GitHub Copilot's coding agent (when an issue is assigned
to `@copilot`) and by other agent runners that follow the
[agents.md convention](https://agents.md). If you are a human, the canonical
contributor docs live in [`CONTRIBUTING.md`](./CONTRIBUTING.md) — this file
is the agent-facing **subset** plus a few rules that only matter when nobody
is at the keyboard.

---

## What this repo is

A monorepo that ships **five independent wheels**, four of which share one
low-level wheel:

```
                ┌──────────────────┐
                │   fabric-core    │   auth · paths · enumerate · diagnostics
                │  (coreHelper/)   │   notebook-build helpers
                └────────┬─────────┘
                         │  imported by
        ┌──────┬─────────┼──────────┬────────────┐
        ▼      ▼         ▼          ▼            ▼
   scanner   downloader  splitter   (mpe)      (reporting)
```

- `fabric-scanner`, `fabric-downloader`, `fabric-splitter` depend on
  `fabric-core>=0.1,<1.0` and never import each other.
- `mpeHelper/` and `reportingHelper/` are stand-alone notebooks / scripts
  (not wheels yet) — only touch them when the issue explicitly says so.

## Issue-to-PR ground rules

When you pick up an issue:

1. **Match the issue's `Which helper is affected?` field to exactly one
   package directory.** Do not cross packages unless the issue body says
   so. The bug-report and feature-request templates put this dropdown at
   the top — read it.
2. **Open a draft PR**, never a ready-for-review PR. A human will mark it
   ready.
3. **Title format**: `<scope>: <short description> (closes #<issue>)`,
   where `<scope>` is one of `core`, `scanner`, `downloader`, `splitter`,
   `mpe`, `reporting`, `repo`, `ci`. Example:
   `scanner: handle 404 from /admin/items (closes #123)`.
4. **One concern per PR.** Refactors that aren't required by the issue go
   in a follow-up PR.
5. **Always run the local test suites you touched** before declaring
   complete (see "Verification" below). If you can't run tests, say so
   in the PR description.

### Labels you must honor

| label | meaning |
|---|---|
| `agent-ok` | safe for autonomous PR drafting |
| `agent-no` | sensitive — humans only, do not pick up |
| `needs-design` | requires a design discussion in comments before code |
| `good-first-issue` | implementation should fit in <300 LOC, mostly mechanical |

Do not pick up an issue labelled `agent-no` or `needs-design`. If an issue
has neither `agent-ok` nor `agent-no` and is in your scope, leave a comment
asking for `agent-ok` instead of opening a PR.

### Maintainer setup: `COPILOT_ASSIGN_TOKEN` (one-time)

The `auto-assign-copilot.yml` workflow auto-assigns `@copilot` to issues that
carry `agent-ok`. To work, it requires a repo secret named
`COPILOT_ASSIGN_TOKEN` — a **fine-grained personal access token** scoped to
this repo with `Issues: Read and write` permission. GitHub's default
`GITHUB_TOKEN` (a GitHub App installation token) cannot assign Bot actors
like Copilot. Without the PAT, the workflow fails fast with an explicit
error pointing at the workflow file's header for setup steps:
[`.github/workflows/auto-assign-copilot.yml`](.github/workflows/auto-assign-copilot.yml).

## Architectural invariants (CI will fail if you break these)

### 1. Dependency direction

- `fabric-core` MUST NOT import from any consumer.
- The three consumer wheels (`scanner` / `downloader` / `splitter`) MUST
  NOT import each other.

Each package ships `tests/unit/test_no_cross_imports.py` (an AST walker)
that enforces this in CI. If you copy code from one consumer into another,
**stop** and move it to `fabric-core` behind a small interface instead.

### 2. `fabric-core` is the deduplication target

When in doubt:

| Belongs in `fabric-core` | Stays in the consumer |
|---|---|
| Token acquisition (any audience) | Consumer-specific config dataclasses |
| OneLake / Fabric REST URL math | Spark `StructType` schemas |
| Workspace + item enumeration | Persist / SQL rollup logic |
| Endpoint-probe diagnostics | Output-path builders |
| Notebook-build serialization helpers (`_md`, `_code`, `_slug`) | Per-package `scripts/build_notebook.py` cell template |

### 3. Mock-patching `fabric-core` re-exports

Consumer `auth.py` re-exports symbols from `fabric_core.auth`. Tests must
patch the **definition** module, not the re-export:

```python
# ❌ NO — patches a name the consumer re-exports but doesn't call
with mock.patch("fabric_scanner.api.auth._from_env", ...): ...

# ✅ YES
with mock.patch("fabric_core.auth._from_env", ...): ...
```

`get_token` resolves `_from_env` in `fabric_core.auth`'s namespace, so
patching the re-export is a silent no-op.

### 4. Generated notebooks must be in sync with their builder

`scannerHelper/notebooks/*.ipynb` and `downloaderHelper/notebooks/*.ipynb`
are generated by each package's `scripts/build_notebook.py`. CI re-runs
the builder and `git diff --exit-code`s `notebooks/`. When you touch a
cell template, regenerate the notebook and commit the result:

```bash
cd <pkg>Helper && python scripts/build_notebook.py
```

Cell IDs are SHA-1 hashes of cell source, so unchanged templates produce
byte-identical output.

### 5. Lower-bound bumps when adopting new `fabric-core` APIs

If you call a function added in `fabric-core` 0.1.5 from a consumer, bump
that consumer's lower bound:

```toml
# scannerHelper/pyproject.toml
dependencies = ["fabric-core>=0.1.5,<1.0", ...]
```

Without this, `pip install fabric-scanner` could resolve to a stale core.

## Verification

Before declaring a PR complete, run **at minimum**:

```bash
# The package(s) you touched
cd <pkg>Helper && pytest tests/unit -q

# If you touched fabric-core, also run all three downstream test suites:
cd coreHelper       && pytest tests/unit -q
cd scannerHelper    && pytest tests/unit -q
cd downloaderHelper && pytest tests/unit -q
cd splitterHelper   && pytest tests/unit -q

# If you touched a notebook cell template, regenerate and verify clean diff:
cd <pkg>Helper && python scripts/build_notebook.py
git diff --exit-code -- notebooks/

# If you touched mpeHelper/terraform/:
cd mpeHelper/terraform && terraform fmt -check -recursive && terraform validate
```

Put a checked-off list of "what I ran and what passed" in the PR description.
If a test couldn't run in your environment, say which and why.

## Coding conventions

- **Python 3.10+** syntax (`X | Y`, `tuple[str, ...]`, etc.) — every wheel
  pins `requires-python = ">=3.10"`.
- **Public API surfaces** (the `__init__.py` `__all__` and dataclass
  fields) are stable from each wheel's first minor release. Adding fields
  to a frozen dataclass with defaults is fine; removing or renaming is a
  breaking change and needs an issue discussion first.
- **Frozen dataclasses for config**, every knob has a sensible default,
  validate in `__post_init__`. Mirror existing patterns in
  `coreHelper/src/fabric_core/`, `scannerHelper/src/fabric_scanner/config.py`,
  etc.
- **Lazy imports** for `pyspark`, `notebookutils`, `aiohttp`. The engine
  half of every wheel must import cleanly on a vanilla Python install
  (no extras). The `[spark]` / `[api]` / `[notebook]` extras gate the
  heavy deps.
- **No `print` in library code.** Use `logging.getLogger(__name__)` and
  let the caller configure handlers. The orchestration notebooks are the
  only place `print(...)` / `display(...)` is OK.
- **No secrets in commits.** Use the existing `azure.identity` /
  `notebookutils.credentials.getToken` plumbing. Never hardcode tokens,
  connection strings, or workspace GUIDs — tests use synthetic values.
- **Type hints everywhere.** `from __future__ import annotations` at the
  top of every file.
- **Don't add new top-level dependencies** without explicit approval in
  the issue. The wheel surface is currently zero non-stdlib deps in the
  pure engine; opt-in heavy deps live behind extras.

## Files an agent SHOULD NOT touch without explicit instruction

| path | why |
|---|---|
| `.github/workflows/main.yml` | release pipeline; one bad change ships a broken wheel |
| `.github/workflows/release-package.yml` | same |
| Any `pyproject.toml` `[project] version` field | versions are bumped by `release-package.yml`, never by hand |
| `*/src/*/_version.py` | same — touched only by the release job |
| `LICENSE`, `CODEOWNERS` | legal / governance |
| `mpeHelper/terraform/` (state, vars files) | infra; needs operator review |

If an issue genuinely needs one of these changed, **stop and ask in the
issue thread** instead of opening a PR.

## What "done" looks like

A PR is ready for human review when:

- [ ] Issue's helper-scope is respected; one concern per PR.
- [ ] All affected packages' `pytest tests/unit -q` is green.
- [ ] If notebook cell templates changed: `scripts/build_notebook.py`
      re-ran and produces a clean `git diff`.
- [ ] No new `[project] dependencies` entries (or they're documented in
      the PR body and the issue thread approved them).
- [ ] `fabric-core` dependency lower-bound updated if you call a new core
      API.
- [ ] PR description has a "what I ran / what passed" block and links
      back to the issue with `Closes #N`.
- [ ] PR is **draft**, not ready-for-review.
